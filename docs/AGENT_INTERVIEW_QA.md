## 2、项目介绍，重点介绍与 AI Agent 相关的项目经验

### 参考答案

我最近重点研究的 Agent 项目是 `mooc-manus-java`。这个项目我把它理解成一个基于 **Java 21 + Spring Boot + Spring AI** 的 Agent 后端系统，它的核心目标不是简单问答，而是让模型能够针对复杂任务先规划、再逐步执行，并且在执行过程中调用工具、把过程通过 SSE 实时推给前端。

从架构上看，它比较典型地分成几层：

- **接口层**：接收 HTTP / SSE 请求，比如 `SessionController` 负责 `/sessions/{id}/chat` 这类接口；
- **服务编排层**：比如 `ChatService` 负责区分“发送新消息”和“断线续流”两条路径；
- **Agent 执行层**：`AgentRunner` 驱动真正的 AI 执行，把事件流统一交给事件总线处理；
- **Flow 层**：`SpringAIPlannerReActFlow` 负责 Planner 和 ReAct 的状态流转；
- **基础设施层**：包括 Redis Stream、SSE、JPA 持久化、Sandbox、工具注册等。

这个项目里我觉得最值得讲的几个点是：

### 第一，Planner + ReAct 的组合执行流

它不是让一个 Agent 直接从头跑到尾，而是先由 Planner 生成任务规划，再由 ReAct Agent 按步骤执行，每执行完一步再更新规划，直到总结输出。`SpringAIPlannerReActFlow` 里把状态拆成了 `IDLE -> PLANNING -> EXECUTING -> UPDATING -> SUMMARIZING -> COMPLETED`，这个设计很适合讲“为什么 Agent 不能只是一个 while loop，而要有显式状态机”。

### 第二，SSE 流式交互 + Redis Stream 断线续流

前端不是等整个任务结束后一次性返回，而是通过 SSE 持续收到 `plan`、`step`、`tool`、`message`、`wait`、`done` 这些事件。这里 `SessionController` 用虚拟线程承载阻塞流程，`ChatService` 里把用户消息和 Agent 事件写入 Redis Stream，这样前端如果断开，再带 `event_id` 过来就可以从 Stream 继续读，不会把中间过程丢掉。

### 第三，工具调用不是“黑盒”，而是有标准事件流和记忆格式

在 `BaseSpringAIAgent` 里，模型输出的工具调用会先转成 assistant message 里的 `tool_calls`，执行完工具后再用 `role=tool` 的消息回填到 memory 里，同时往前端发 `ToolEvent`。这说明它不是简单把工具藏在框架内部，而是把工具调用过程外显成事件，这对调试、回放和可观测性都更友好。

### 第四，短期记忆、回滚和压缩机制

这个项目的记忆不是只存最终答案，而是会按 `sessionId + agentName` 存原始消息轨迹。遇到 `message_ask_user` 这种等待用户输入的场景，还会通过 `rollBack` 修正消息链，避免 assistant 的工具调用之后直接接一个 user message 导致格式错误。另外执行过程中还会做 `compactMemory()`，把浏览器这类重量级 observation 压缩掉，减少上下文污染和 token 消耗。

如果让我用一句话总结这个项目，我会说：

> 它让我真正把 Agent 看成一个“有状态、有事件、有工具、有记忆的后端系统”，而不是一个简单的大模型接口封装。

### 可扩展点

如果面试官问“你在这里面具体收获了什么”，可以答：

> 我最大的收获是把以前做业务后端的很多方法迁移到了 Agent 上，比如状态机、事件驱动、失败兜底、幂等、断点续跑、能力解耦。这些在传统后端里很常见，但到了 Agent 里一样重要，甚至更重要。

## 3、多模态大模型的具体结构是什么样的？请详细描述视觉编码器和语言模型的衔接方式

### 参考答案

多模态大模型如果只看“图像 + 文本”这个最常见场景，核心结构通常可以拆成三块：

1. **视觉编码器（Vision Encoder）**：负责把图片转成视觉特征；
2. **模态对齐层 / 投影层（Projector / Adapter / Q-Former）**：负责把视觉特征映射到语言模型能消费的表示空间；
3. **语言模型（LLM）**：负责统一做推理、生成和工具调用。

如果更详细一点，流程一般是这样的：

### 第一步，图像先经过视觉编码器

常见做法是用 ViT 或 CLIP Vision Encoder，把图像切成 patch，然后编码成一串视觉 token。这个阶段模型拿到的不是“猫”“桌子”这种语义词，而是一组高维向量表示。

### 第二步，做视觉特征到语言空间的对齐

这一步是多模态模型的关键，因为视觉编码器输出的 embedding 空间和 LLM 的 token embedding 空间并不天然一致。工程里常见有三种衔接方式：

#### 方式 1：线性投影 / MLP Projector

最简单的方式，就是把视觉 encoder 输出的向量，通过一个线性层或 MLP 映射到 LLM hidden size 对应的维度，然后把这些映射后的向量当作“伪 token”拼到文本 token 前面或中间。

优点是结构简单、代价低；缺点是表达能力有限。

#### 方式 2：Q-Former

像 BLIP-2 这种方法会引入一个 Q-Former，用少量 query token 去从视觉特征里抽取最有信息量的部分，再把这些 query 的输出送给 LLM。它相当于在视觉 encoder 和 LLM 之间加了一层压缩和对齐模块。

这种方式的好处是能减少直接把大量视觉 token 全部喂给 LLM 带来的上下文成本。

#### 方式 3：Cross-Attention 融合

还有一些模型是在 LLM 内部引入 cross-attention，让语言 token 在生成过程中去 attend 视觉特征，而不是简单拼接。这种方式融合得更深，但改造成本也更高。

### 第三步，把视觉 token 和文本 token 一起交给 LLM

对 LLM 来说，最核心的事情是：它最终消费的是一段统一的 token 序列或者统一的 hidden states。也就是说，多模态模型本质上是把非文本信息转成一种可被 LLM 理解的“语言空间表示”，然后仍然交给语言模型完成后续推理。

### 我对这个问题的理解重点

我会把它总结成一句话：

> 多模态模型的关键，不在于“有个视觉编码器”这么简单，而在于如何把视觉信号压缩、筛选、对齐成语言模型可以稳定消费的 token 表示。

如果面试官继续追问训练层面，可以继续讲：

- 一般先做视觉-语言对齐训练；
- 再做 instruction tuning，让模型学会根据图文输入做问答、描述和推理；
- 更高阶的还会加入 OCR、区域 grounding、视频时序建模等能力。

### 贴合你当前现状的说法

> 这个方向我目前的理解主要来自系统学习和项目拆解，不是自己从零训过多模态模型。但我能比较清楚地讲清楚视觉 encoder、projector、LLM 三者怎么衔接，以及不同衔接方式的工程权衡。

## 4、ReAct 框架的工程实现细节，消息格式如何设计？tool response 应该用什么角色传回？为什么？

### 参考答案

ReAct 从工程上看，不是一个抽象理念，而是一套比较严格的消息协议。最小闭环通常包括四类消息：

- `system`：系统约束，定义 Agent 身份、行为边界、工具使用原则；
- `user`：用户输入或外部新指令；
- `assistant`：模型的思考结果、普通回复，或者带 `tool_calls` 的工具调用意图；
- `tool`：工具执行后的 observation，也就是工具结果回传。

### tool response 应该用什么角色？

标准答案是：**应该用 `tool` 角色返回，而不是 assistant 或 user。**

原因有三个：

#### 第一，语义边界清晰

`assistant` 代表模型自己生成的内容，`tool` 代表外部环境返回的 observation。两者语义完全不同。如果把工具结果也塞进 assistant，会混淆“模型推理”和“环境反馈”的边界。

#### 第二，方便模型正确续推理

在 function calling 或 tool calling 协议里，模型先输出一个 assistant message，里面带 `tool_calls`；工具执行完后，下一条消息应该是 `tool` role，并通过 `tool_call_id` 绑定到对应调用。这样模型能知道：

- 这是哪个工具的返回；
- 这不是新的用户输入；
- 这是我下一轮推理要基于的 observation。

#### 第三，便于审计、回放和训练

如果后面要做 trace 回放、SFT 样本构造、错误定位，`assistant` / `tool` / `user` 的边界必须清楚。否则数据会乱。

### 结合 `mooc-manus-java` 的工程实现来讲

这个项目里，`BaseSpringAIAgent` 的做法是很标准的：

1. 模型返回 `AssistantMessage`，里面可能带 `tool_calls`；
2. 系统执行工具；
3. 工具结果回写到 memory 时，明确保存成：
   - `role = tool`
   - `tool_call_id = xxx`
   - `function_name = xxx`
   - `content = toolResultJson`
4. 同时再构造 `ToolResponseMessage` 放回当前消息链，让模型继续下一轮推理。

所以如果面试官问我“为什么你这么设计”，我会说：

> 因为 ReAct 的核心就是把 Think / Act / Observe 三段拆开，assistant 负责 Think + Act 的声明，tool 负责 Observe 的回传。如果 tool response 不用 `tool` role，整个轨迹的语义就会被破坏。

### 一个推荐的消息格式示意

```json
[
  {"role": "system", "content": "你是一个任务执行智能体"},
  {"role": "user", "content": "帮我查一下某商品最近的补货建议"},
  {
    "role": "assistant",
    "content": "",
    "tool_calls": [
      {
        "id": "call_1",
        "type": "function",
        "function": {
          "name": "query_inventory",
          "arguments": "{\"sku\":\"123\"}"
        }
      }
    ]
  },
  {
    "role": "tool",
    "tool_call_id": "call_1",
    "content": "{\"stock\":12,\"in_transit\":30}"
  },
  {
    "role": "assistant",
    "content": "基于当前库存和在途量，我建议..."
  }
]
```

### 可扩展点

如果面试官追问“那 `toolresponse` 能不能走 assistant？”，可以答：

> 不建议。除非是你做一个内部非标准协议，否则从通用模型 API、后续训练数据构造、可回放性和调试体验来看，都会明显变差。

## 5、Agentic CPT、SFT、RL 三阶段训练流程分别是什么？为什么 SFT 时要 mask observation tokens？

### 参考答案

如果把 Agent 模型训练分成三阶段去理解，一般可以概括成：

- **CPT（Continued Pretraining）**：补领域知识和 Agent 相关语料；
- **SFT（Supervised Fine-Tuning）**：让模型学会正确的 Agent 轨迹格式；
- **RL（Reinforcement Learning）**：进一步优化策略质量、工具使用效率和长程决策。

### 第一阶段：Agentic CPT 是什么

CPT 本质上是继续预训练。目标不是教模型固定答案，而是让它先熟悉 Agent 场景里的语言分布和知识结构。常见语料包括：

- API 文档、工具说明；
- 代码仓库、命令行交互；
- 任务规划数据；
- 网页、知识库、企业内部 SOP；
- tool trace、日志、工单等。

这一步的目标是让模型先“会读懂 Agent 世界里的文本”。

### 第二阶段：SFT 是什么

SFT 更关键，它是让模型学会标准的 Agent 行为轨迹。训练样本一般不是单一问答，而是一条完整 trajectory，比如：

- 用户问题；
- assistant 规划；
- assistant 发起 tool call；
- tool observation；
- assistant 基于 observation 做下一步；
- 最终回答。

也就是说，SFT 要让模型学会：

- 什么时候该调用工具；
- 工具参数怎么组织；
- observation 回来后如何继续推理；
- 什么时候总结；
- 什么时候 ask user。

### 第三阶段：RL 是什么

RL 的目标不是让模型更会“模仿”，而是让它更会“决策”。比如可以用 reward 去优化：

- 是否完成任务；
- 工具调用是否更少、更准；
- 是否减少死循环；
- 是否遵守安全约束；
- 最终结果是否更符合用户目标。

在 Agent 场景下，RL 更像是在优化一个策略，而不只是优化一句话的回答质量。

### 为什么 SFT 时要 mask observation tokens？

这是一个很关键的点。我的理解是：**observation tokens 是环境返回，不是模型应该学习去预测的目标，所以通常要 mask。**

原因主要有三点：

#### 1. observation 不是模型决策产物

工具返回结果是外部系统给的，比如搜索结果、数据库查询结果、网页内容。这部分不属于模型“应该输出什么”，而属于环境状态。

#### 2. 避免模型去死记 observation 分布

如果不 mask，模型会被训练成去拟合这些 observation 的具体内容，容易浪费学习能力，甚至出现错误偏向：它更像在背工具输出，而不是学“看到 observation 后如何决策”。

#### 3. 强化“行动策略学习”而不是“环境复述学习”

Agent SFT 更想学的是：

- 在什么 observation 下该怎么继续；
- 什么时候继续调工具；
- 什么时候停止；
- 如何生成最终答案。

所以一般会保留 observation 作为上下文输入，但不把它们作为 loss 目标。

### 可以怎么总结

> CPT 解决“懂不懂 Agent 世界”；SFT 解决“会不会按 Agent 轨迹做”；RL 解决“做得好不好、稳不稳、效率高不高”。而 observation token 在 SFT 里通常 mask，是因为它是环境给的信息，不是模型需要去模仿生成的内容。

### 贴合现状的回答边界

> 训练这块我目前更多是工程视角的理解，不会说自己亲自做过完整大模型训练，但这三阶段的目的、样本结构和为什么 mask observation，我是能讲清楚的。

## 6、Redis 在 AI Agent 系统中可能有哪些应用场景？如何设计缓存策略？

### 参考答案

Redis 在 Agent 系统里我觉得至少有六类典型用途：

### 1. 事件流 / 断线续流

这个是 `mooc-manus-java` 里最直接的应用。项目里 `RedisStreamTask` 用 Redis Stream 承载 task 的 input / output stream，`ChatService` 和 `AgentRunner` 会把事件写到 output stream，前端断线后能基于 `event_id` 继续读取。

### 2. 会话状态缓存

比如 session 是否在运行中、最近一次活跃时间、锁状态等，都可以放 Redis。这样可以减轻数据库热点压力。

### 3. 工具结果缓存

比如搜索结果、知识检索结果、某些低频变更的外部接口数据，可以做短 TTL 缓存，降低成本和延迟。

### 4. 分布式锁 / 并发控制

同一个 session 同时只能有一个 agent run，这种场景如果是多实例部署，可以用 Redis 做分布式锁。

### 5. 限流与配额控制

比如按用户、按会话、按工具维度做 QPS / 并发数 / token 配额限制，Redis 很适合做计数器。

### 6. 短期记忆或中间态缓存

像临时 summary、query rewrite 结果、工具选择候选集、执行中的草稿状态，都可以放 Redis 做中间层。

### 缓存策略怎么设计

我一般会按“数据特征”来分：

#### 第一类：强一致要求高，不轻易缓存

比如资金类、库存实时扣减、核心交易状态，这种不要因为 Agent 场景就滥用缓存。缓存只能做读优化，最终一致性一定要有清晰边界。

#### 第二类：可容忍短暂过期，适合 TTL 缓存

比如：

- 检索结果；
- 用户画像；
- 工具 schema；
- 非强实时知识片段。

TTL 可以按分钟级到小时级设计。

#### 第三类：带参数的结果缓存

比如工具调用结果，可以用“工具名 + 归一化参数 + 版本号”做 key。这样能避免参数顺序不同导致缓存不命中。

#### 第四类：热点 session / 热点 memory

热门会话的最新 summary、最近 N 条对话，可放 Redis 做热缓存，但最终还是要落持久层。

### 我会特别强调的几个缓存原则

1. **缓存 key 要可追踪**，最好能看出 session、tool、tenant、version；
2. **TTL 要按场景分层**，不能全局一个 TTL；
3. **缓存击穿 / 雪崩 / 穿透要预防**，如互斥锁、随机过期、空值缓存；
4. **对强副作用操作不要直接做结果缓存**，特别是写操作工具；
5. **缓存和 memory 不要混为一谈**，缓存偏性能，memory 偏语义与任务连续性。

### 一句话总结

> Redis 在 Agent 系统里不只是做缓存，更重要的是做“状态中介层”，包括续流、分布式锁、限流、热点数据和中间态管理。

## 7、MySQL 索引在什么情况下会失效？使用 LIKE 进行模糊查询时索引什么情况下会失效？

### 参考答案

MySQL 索引失效本质上可以理解成一句话：**优化器判断走索引不划算，或者 SQL 写法让索引没法按有序结构利用起来。**

常见情况有这些：

### 1. 对索引列做函数、计算或隐式类型转换

比如：

- `where date(create_time) = '2026-05-17'`
- `where sku_code + 0 = 123`
- 字符串字段拿数字去比较，触发隐式转换

这样通常会让索引失效。

### 2. 不满足最左前缀原则

复合索引比如 `(a, b, c)`，如果你的条件直接从 `b` 或 `c` 开始，而没有利用 `a`，就很难走好索引。

### 3. 范围查询后面的列利用率下降

比如复合索引 `(a, b, c)`，如果 `a = ? and b > ? and c = ?`，通常 `c` 很难继续有效利用。

### 4. 使用 `!=`、`<>`、`not in`、低选择性条件

这类条件优化器常常认为回表成本高，不如直接扫表。

### 5. `or` 条件一侧没索引

如果 `or` 两边不是都能有效使用索引，整体可能就退化成全表扫描。

### 6. 数据量小或统计信息不准

有时候不是索引真“失效”，而是优化器评估后认为全表扫更便宜。

### LIKE 在什么情况下失效

关键结论：**前导模糊会失效，后缀模糊通常还能用。**

- `like 'abc%'`：一般可以用索引；
- `like '%abc'`：通常不能用索引；
- `like '%abc%'`：通常也不能有效用 B+ 树索引。

因为 B+ 树索引本质上是按前缀有序存储的，只有从左边开始匹配，才能利用“有序”这个特性。

### 面试里可以顺手补一个优化思路

如果业务上经常要做包含匹配：

- 可以考虑倒排搜索，比如 Elasticsearch；
- 或者引入前缀索引、搜索字段冗余；
- 如果是业务编码、SKU、单号，尽量约束查询方式为前缀匹配。

### 贴合你的背景的表达方式

> 在供应链系统里，像单号、商品编码、计划单状态这类字段通常都比较适合做精确或前缀匹配；如果出现大量包含搜索，我会更倾向于把这类需求从 MySQL B+ 树索引转移到搜索引擎处理，而不是强行让数据库扛。

## 8、消息队列在 AI Agent 系统中的作用是什么？为什么不直接通过数据库通信？

### 参考答案

消息队列在 Agent 系统里，核心作用是把“请求提交”和“任务执行 / 事件消费”解耦开。常见价值有这些：

### 1. 削峰填谷

Agent 执行往往比普通接口更慢、更重，有 LLM 调用、工具调用、RAG、浏览器等步骤。如果所有请求都同步压在数据库和接口线程上，系统会很容易抖动。

### 2. 异步化执行

比如用户发起一个复杂任务，接口只负责受理请求，把任务丢进 MQ，后台 worker 再慢慢执行，前端订阅进度事件即可。

### 3. 解耦不同模块

例如：

- 一个模块负责接入层；
- 一个模块负责工具执行；
- 一个模块负责事件分析；
- 一个模块负责观测与告警。

通过 MQ 可以把它们松耦合串起来。

### 4. 提升失败恢复能力

有些任务可以重试、补偿、死信转移，MQ 天然更适合承载这种机制。

### 5. 支持事件驱动架构

Agent 产生的大量事件，比如 plan、tool、message、done、error，都适合走事件流处理，而不是把所有副作用硬写在一个同步事务里。

### 为什么不直接通过数据库通信

数据库当然也能做状态共享，但不适合作为高频过程通信的主通道，原因主要有：

#### 1. 数据库擅长存最终状态，不擅长高频实时消息分发

Agent 事件是过程数据、流数据，不是只有最终结果。数据库更适合存历史和结果，不适合承接大量频繁轮询。

#### 2. 轮询数据库成本高、延迟高

如果前端断线续流或后台 worker 轮询都靠数据库，会产生很多无效查询。

#### 3. 时序和消费语义弱

MQ 更容易处理顺序消费、重试、ack、消费组等语义，数据库要自己补很多机制。

#### 4. 容易把 OLTP 库拖垮

生产里数据库通常是核心业务资源，不应该把 Agent 过程通信也全部堆上去。

### 结合 `mooc-manus-java` 怎么讲

这个项目当前其实是一个很适合面试里讲“过渡架构”的例子：

- 当前聊天请求整体还是同步执行；
- 但 Redis Stream 已经承担了事件记录、断线续流和任务句柄的作用；
- 这说明系统已经具备了向真正异步任务架构演进的基础。

所以我会回答：

> 数据库更适合持久化最终事实，消息队列更适合承载过程中的异步通信和事件流。Agent 系统里如果所有过程通信都靠数据库，会让时延、成本和扩展性都变差。

## 9、RAG 流程中为什么要引入父子索引？BM25 和向量检索如何融合？

### 参考答案

RAG 里引入父子索引，核心是为了同时解决两个问题：**检索精度** 和 **上下文完整性**。

### 为什么需要父子索引

如果文档很长，直接整篇向量化有两个问题：

- chunk 太大，语义被冲淡，检索精度差；
- 一次塞给模型的上下文成本高。

如果 chunk 切得太小，又会出现另一个问题：

- 检索到了局部片段，但缺少上下文，模型理解不完整。

所以工程里常见做法是：

- **子块索引（child chunks）**：切得比较细，用来提升召回精度；
- **父块索引（parent document / parent chunk）**：检索到子块后，再回捞父块，让模型看到更完整的上下文。

### 这样做的价值

- 子块负责“找到相关位置”；
- 父块负责“补足语义背景”；
- 最后喂给模型的是更完整但仍然受控的内容。

### BM25 和向量检索怎么融合

BM25 和向量检索各有优势：

#### BM25 的优势

- 关键词匹配强；
- 对专有名词、型号、编码、术语特别有效；
- 可解释性好。

#### 向量检索的优势

- 语义召回强；
- 能处理同义改写、自然语言问法；
- 对长文本语义相关性更敏感。

所以实际工程里通常不会二选一，而是做 **Hybrid Search**。

### 典型融合方式

#### 方式 1：并行召回 + RRF 融合

BM25 和向量检索各自召回 topK，再用 **RRF（Reciprocal Rank Fusion）** 之类的方法做融合排序。

这是很常见、也很稳妥的方法。

#### 方式 2：BM25 粗召回，向量重排

先用关键词筛一批候选，再用向量相似度或 cross-encoder 做 rerank。

#### 方式 3：向量召回，BM25 规则补强

对于包含 SKU、订单号、接口名等强关键词问题，可以用 BM25 做强约束。

### 面试里建议这么总结

> 父子索引解决的是“检索粒度”和“上下文完整性”的矛盾；BM25 + 向量融合解决的是“关键词精确匹配”和“语义召回”的互补问题。线上一般会做混合召回，再结合 rerank 提升最终质量。

### 结合你的背景补一句

> 如果是在供应链场景，我会特别重视 BM25，因为很多问题其实带有明确的编码、单号、规则名、仓库名，这些强关键词用 BM25 很有价值，而向量更多是补自然语言语义召回。

## 10、Agent 的记忆机制怎么设计？短期记忆和长期记忆分别如何实现？

### 参考答案

我理解 Agent 的记忆不是一个单点能力，而是一套分层设计。最常见的就是拆成 **短期记忆** 和 **长期记忆**。

### 1. 短期记忆是什么

短期记忆主要服务当前会话，用于保证这轮任务能连续执行。它一般包括：

- 最近几轮对话；
- 当前 plan；
- 当前 step 的状态；
- 最近的 tool observation；
- 当前待确认的信息。

在 `mooc-manus-java` 里，我会把 `AgentMemoryStore` 里按 `sessionId + agentName` 存的消息轨迹理解成短期记忆的一部分。它存的不是摘要，而是较原始的轨迹，方便 ReAct 继续推理。

### 2. 长期记忆是什么

长期记忆更像稳定知识和用户偏好的沉淀，常见包括：

- 用户偏好；
- 业务规则；
- 历史成功案例；
- 知识文档；
- 历史总结后的结构化 fact。

长期记忆一般不会每轮都全量塞给模型，而是按需检索。

### 短期记忆怎么实现

短期记忆常见三种形式：

#### 形式 1：最近窗口

保留最近 N 轮 message / tool trace。简单直接，但容易超上下文。

#### 形式 2：summary + recent window

把旧内容做摘要，只保留最近几轮原始消息，这是线上很常见的折中方案。

#### 形式 3：显式状态槽位

比如把“当前任务目标、待确认信息、已完成步骤、用户约束”单独结构化存下来，而不是全靠自然语言对话恢复。这个在复杂 Agent 里很重要。

### 长期记忆怎么实现

我一般会把长期记忆拆成两类：

#### 1. 语义型长期记忆

放向量库，适合做“相似任务”“历史经验”“文档知识”的召回。

#### 2. 结构化长期记忆

放数据库或 KV，适合用户偏好、白名单、业务配置、常用模板等强结构信息。

### 我认可的一套落地方案

- **短期记忆**：当前会话轨迹 + 当前 summary + 状态槽位；
- **长期记忆**：向量检索的经验库 + 结构化用户画像 / 规则库；
- **读取策略**：不是全读，而是按任务类型和当前步骤动态拼接。

### 结合 `mooc-manus-java` 的可讲点

这个项目里已经有两个很典型的记忆设计点：

1. `AgentMemoryStore`：保存 agent 轨迹；
2. `compactMemory()`：对重量级 observation 做压缩，避免上下文无限膨胀。

所以我会回答：

> Agent 的记忆不能等同于聊天记录。短期记忆重点解决当前任务连续性，长期记忆重点解决跨任务复用。工程上要把原始轨迹、摘要记忆和结构化状态分开设计。

## 11、上下文超出限制时如何处理？滑动窗口和动态摘要的区别是什么？

### 参考答案

当上下文超限时，最常见的处理方式有四类：

1. **滑动窗口**：只保留最近 N 轮；
2. **动态摘要**：把旧内容压缩成 summary；
3. **检索回填**：只把相关历史按需召回；
4. **结构化状态抽取**：把关键事实从自然语言里抽出来单独保存。

### 滑动窗口和动态摘要的区别

#### 滑动窗口

滑动窗口的核心思路是：**旧内容直接丢掉，只保留最近上下文。**

优点：

- 简单；
- 实现成本低；
- 时延稳定。

缺点：

- 重要信息可能被截断；
- 长任务容易遗忘早期约束。

#### 动态摘要

动态摘要的思路是：**旧内容不直接丢，而是压缩成一段 summary，继续保留关键事实。**

优点：

- 更适合长任务；
- 可以保留用户约束、目标和历史决策。

缺点：

- 摘要本身可能失真；
- 需要额外模型开销；
- 摘要策略设计不好会越压越偏。

### 我的建议

线上一般不要只用一种，而是：

- 最近几轮用原始窗口；
- 更早内容做动态摘要；
- 特别关键的信息抽成结构化 memory；
- 需要时从长期记忆检索补回。

### 结合项目怎么讲

`mooc-manus-java` 当前没有特别复杂的长期 memory pipeline，但已经有 `compactMemory()` 这种“对 observation 做压缩”的思路，这本质上就是上下文治理的一个工程点。

所以我的回答会偏工程：

> 滑动窗口解决的是“快速截断”，动态摘要解决的是“保留关键语义”。如果任务链较长，只用窗口会忘事，只用摘要又容易失真，最稳妥的是窗口、摘要和结构化状态结合。

## 12、手写算法：实现反转链表

### 参考答案

这题最经典的写法是迭代，核心就是三个指针：`prev`、`cur`、`next`。

```java
class ListNode {
    int val;
    ListNode next;
    ListNode(int val) { this.val = val; }
}

public class Solution {
    public ListNode reverseList(ListNode head) {
        ListNode prev = null;
        ListNode cur = head;
        while (cur != null) {
            ListNode next = cur.next;
            cur.next = prev;
            prev = cur;
            cur = next;
        }
        return prev;
    }
}
```

### 讲解思路

- `prev` 表示已经反转好的前半段；
- `cur` 表示当前节点；
- 每次先保存 `next`，防止链断；
- 然后把 `cur.next` 指向 `prev`；
- 最后整体往前推进。

### 复杂度

- 时间复杂度：`O(n)`
- 空间复杂度：`O(1)`

### 面试时可补一句

> 如果面试官继续追问，我也可以再写递归版本，但迭代版本更稳，也更符合生产代码习惯。

## 2、详细描述你最近参与的 AI Agent 项目，包括架构设计和技术选型

### 参考答案

我最近重点拆解和实践的是 `mooc-manus-java`。如果从“一个请求怎么走完”来描述，它的主链路大致是这样的：

1. 前端通过 `/sessions/{id}/chat` 发起请求；
2. `SessionController` 建立 SSE，并用虚拟线程承接阻塞式 Agent 执行；
3. `ChatService` 判断是新消息还是续流；
4. 新消息场景下，会先校验 session 状态、记录用户消息、准备 Task / Sandbox；
5. 再交给 `AgentRunner` 驱动具体的 AgentStrategy；
6. 底层由 `SpringAIPlannerReActFlow` 完成 Planner + ReAct 的执行闭环；
7. 执行过程中产生的 plan / step / tool / message / wait / done 等事件，一边进入 Redis Stream，一边通过事件总线处理持久化和 SSE 推送。

### 为什么这样选型

#### 1. 选 Java 21 + Spring Boot

因为我本身是 Java 后端背景，这样技术栈和工程习惯最一致，而且虚拟线程对于这类“LLM 调用耗时长、工具调用阻塞明显”的场景很友好。

#### 2. 选 Spring AI

原因不是说它能力最强，而是它和 Spring 生态融合自然，适合 Java 项目快速搭 Agent 能力。对于我来说，重点不是框架本身，而是把 Planner、ReAct、Tool Callback、Memory 这些结构先理清楚。

#### 3. 选 SSE 而不是 WebSocket

因为当前核心需求是服务端单向持续推事件，SSE 就足够，而且接入简单、代理兼容性好。`SessionController` 里直接通过 `SseEmitter` 做流式返回，这个路径很清楚。

#### 4. Redis Stream 用于续流与任务输出

当前项目不是完全异步任务架构，但 Redis Stream 已经承担了过程事件通道和断点续流的职责。这一点我觉得很像很多真实系统的演进路线：先把同步链路跑通，再逐步往后台异步任务架构演进。

### 架构亮点

我认为有四个点比较能体现 Agent 工程设计：

1. **显式状态流转**：`SpringAIPlannerReActFlow` 不是黑盒调用，而是可讲解的状态机；
2. **消息协议清晰**：assistant / tool / user 角色边界清楚；
3. **事件流外显**：plan、step、tool、done 都是独立事件，便于调试和回放；
4. **异常兜底**：`AgentRunner` 在出错时会补发 `ErrorEvent` 并把 session 状态归位，避免会话锁死。

### 这题最稳妥的收口方式

> 这个项目我更愿意定义成一个“深度拆解并具备工程实践价值的 Agent 项目”，它让我对 Agent 后端的关键问题——编排、状态、工具、记忆、流式输出、续流和错误恢复——建立了系统理解。

## 3、你的 Agent 如果出现死循环怎么办？异常处理机制如何设计？

### 参考答案

Agent 死循环是必须按工程问题来看待的，不能指望模型自己永远聪明。我的处理思路一般分四层。

### 第一层：在 Agent Loop 上设硬边界

比如：

- 最大迭代次数；
- 单步超时；
- 单工具最大重试次数；
- 同一工具 / 同一参数重复调用阈值。

`mooc-manus-java` 里 `BaseSpringAIAgent` 就有 `maxIterations` 和 `maxRetries` 这类限制，这就是最基础的防死循环手段。

### 第二层：识别“重复模式”

比如出现下面这些信号时，我会主动打断：

- 连续多轮调用同一个工具且参数几乎一样；
- summary 不前进，step 状态一直不变化；
- 连续报同类错误；
- message_ask_user 和工具调用之间来回振荡。

这类检测可以通过 trace 或执行日志做规则判断。

### 第三层：异常分级处理

我通常会把异常分成三类：

#### 1. 可重试异常

比如网络抖动、短暂超时、429，这类可以按退避策略重试。

#### 2. 可恢复异常

比如某个工具失败，但可以换备选工具或降级回答。这时 Agent 不一定要整体失败。

#### 3. 致命异常

比如消息协议损坏、状态不一致、上下文无法继续，这种要快速终止，并把 session 状态归位。

### 第四层：状态兜底

这是最容易被忽略的。即使 Agent 出错，也不能让会话永远停在 RUNNING。`AgentRunner` 里就做了这件事：捕获异常后发 `ErrorEvent`，并把 session 状态从 RUNNING 改回 COMPLETED。这种设计非常重要，否则前端会看到这个会话永远不可再次发消息。

### 如果要更进一步，我会加什么

- watchdog 定时检查超长运行会话；
- 死循环特征告警；
- trace 回放；
- 人工接管入口；
- 对 wait / done / error 这类终止事件做强状态校验。

### 一句话总结

> Agent 出现死循环，不应该只靠 prompt 修，而应该从“迭代边界、重复模式识别、异常分级、状态归位”四个层面一起做。

## 4、工具库有上百个工具时，如何让模型快速准确地选择工具？

### 参考答案

工具多了以后，问题不只是 prompt 太长，更是选择精度会急剧下降。所以我不会把所有工具平铺给模型，而是做分层和路由。

### 第一步：工具分层分类

先按工具性质拆层：

- 文件类；
- Shell / 执行类；
- 搜索类；
- 浏览器类；
- 业务类；
- 高风险写操作类；
- 外部协议类，如 MCP / A2A。

在 `mooc-manus-java` 里，`ToolCallbackRegistry` 实际上就已经有这种组织思路，至少把文件、Shell、浏览器、搜索、消息、MCP、A2A 分开了。

### 第二步：做前置路由，而不是全量暴露

我更倾向于两阶段：

1. 先做 tool routing，选出一个小集合；
2. 再把这部分工具暴露给模型做最终选择。

这个 router 可以是：

- 规则路由；
- 小模型分类；
- embedding 召回；
- 混合方式。

### 第三步：优化工具 schema

工具选不准，很多时候不是模型笨，而是 schema 太差。一个好的工具定义至少包括：

- 工具名要稳定且可读；
- 描述要说明“什么时候用 / 不该什么时候用”；
- 参数要清晰、有约束；
- 最好给出反例和边界说明。

### 第四步：做工具使用反馈闭环

比如统计：

- 工具被选中的准确率；
- 工具调用后失败率；
- 实际完成任务的贡献度。

长期看可以把这些数据反过来优化工具描述和路由策略。

### 我的答案会这样收口

> 工具数上百时，关键不是让模型“直接从 100 个里猜”，而是先做工具治理：分类、路由、精简暴露、优化 schema，再结合调用反馈持续调优。模型只是最后一跳，不是第一跳。

## 5、Agent 执行过程中遇到工具调用失败（如支付接口超时）如何处理？

### 参考答案

我会按“是否有副作用”和“是否可恢复”来处理。

### 第一类：无副作用工具失败

比如搜索超时、网页抓取失败、知识检索失败，这类通常可以：

- 重试；
- 切换备选数据源；
- 降级成部分回答；
- 告知用户当前结果不完整。

### 第二类：有副作用工具失败

比如支付、下单、删除、审批，这类必须更谨慎。我的处理原则是：

1. **先保证幂等**：避免超时后重复执行；
2. **有事务号 / 请求号**：方便查状态；
3. **不要把“超时”直接等同于“失败”**：可能只是调用方没等到结果；
4. **必要时做补偿或人工确认**。

### 如果是支付接口超时，我会怎么设计

- 请求发起前生成业务幂等号；
- 工具层统一封装重试，但只对安全可重试错误重试；
- 超时后先查支付结果，而不是直接再次支付；
- 如果状态不确定，进入“待确认”状态，而不是让 Agent 自己盲目重试；
- 对用户回复时要明确“正在确认处理结果”，而不是瞎承诺成功或失败。

### 和 Agent 结合的关键点

Agent 不能被放任在高风险操作上自由发挥，所以我会加两层：

1. **高风险工具白名单 + 审批机制**；
2. **执行前 dry-run / 参数校验 / 二次确认**。

### 结合 `mooc-manus-java` 可以怎么讲

`BaseSpringAIAgent.executeToolCallback()` 里已经有基础的重试逻辑，但如果放到真正高风险业务里，这还不够。我会在面试里明确表达：

> 对高风险写操作，不能只做“工具失败重试”这么简单，必须引入幂等、状态查询、补偿和人工兜底。Agent 只是决策和编排层，最终落执行时要服从业务安全边界。

## 6、长上下文对话中，如何让 Agent 不忘记关键信息？除了向量检索还有什么方法？

### 参考答案

让 Agent 不忘关键信息，不能只靠向量检索，因为很多重要信息并不是“语义相似”问题，而是“任务状态和约束”问题。

除了向量检索，我更看重下面几种方法：

### 1. 结构化槽位记忆

把关键事实抽出来单独存，例如：

- 用户目标；
- 已确认约束；
- 禁止事项；
- 当前步骤；
- 已完成结果；
- 需要用户补充的信息。

这种方式比纯聊天记录稳定得多。

### 2. 动态摘要

把旧对话压成 summary，但 summary 不应只是自然语言流水账，而应尽量抽成“目标 / 约束 / 决策 / 未决事项”。

### 3. 事件日志 / 状态机

复杂 Agent 尤其适合把核心过程抽成事件和状态，而不是完全依赖 LLM 从自然语言恢复。像 `PlanEvent`、`StepEvent`、`WaitEvent` 这种就是很好的外部状态锚点。

### 4. 最近窗口 + 关键事实拼装

当前轮 prompt 里，我不会只放最近对话，还会拼入：

- 当前 plan；
- 当前 step；
- 最新 summary；
- 强约束 facts。

### 5. 用户画像 / 偏好配置

有些“别忘记”的信息不是聊天内容，而是长期偏好，比如默认语言、输出格式、常用工具或业务偏好，这类应该走结构化存储。

### 总结一句

> 向量检索更适合找“相关知识”，但让 Agent 真正不忘关键信息，更有效的是结构化状态、摘要、事件日志和最近窗口的组合。

## 7、Tree of Thoughts 在线上系统中能用吗？如何平衡成本和效果？

### 参考答案

能用，但我认为 **不能默认全量用**，而应该做条件化使用。

Tree of Thoughts 的本质，是让模型在多个思路分支上探索，然后再评估或回溯。这对复杂推理题、规划问题、约束求解类任务很有帮助，但线上代价也很明显：

- token 成本高；
- 时延高；
- 可控性变复杂；
- 分支过多时效果未必线性提升。

### 线上怎么平衡

我的思路一般是：

#### 1. 只在高价值、复杂任务上触发

比如：

- 复杂规划；
- 多约束决策；
- 高成本错误场景；
- 需要显式比较多个方案的任务。

简单问答不要上 ToT。

#### 2. 限制分支数和深度

比如：

- breadth 设成 2~3；
- depth 不超过 2~3 层；
- 中间用启发式规则剪枝。

#### 3. 先粗后细

先用普通 CoT / ReAct 试一轮，只有置信度低或失败时，再升级为 ToT。

#### 4. 配合外部约束评估

某些分支好不好，不一定让模型自己评。可以结合规则引擎、打分器、执行反馈做评价。

### 我会怎么回答面试官

> Tree of Thoughts 可以在线上用，但更适合作为“高复杂度任务的升级策略”，而不是默认主路径。真正的平衡点在于：分层触发、限制搜索空间、外部评估和成本可控。

## 8、如果 Agent 的决策出错导致数据误删，系统设计上如何防范？

### 参考答案

这个问题本质上不是“如何让模型更聪明”，而是“如何让系统把风险关住”。我一般会从五层来设计。

### 第一层：权限隔离

高风险工具不能默认对所有 Agent 开放，要按角色、租户、环境、任务类型做白名单控制。

### 第二层：读写分离

默认给 Agent 的大部分能力应当是读权限。写权限要少、要窄、要有边界。

### 第三层：二次确认 / 审批

像删除、支付、批量更新这种操作，我会要求：

- 先给出 dry-run 结果；
- 展示影响范围；
- 要么用户确认，要么人工审批；
- 确认后才真正执行。

### 第四层：软删除 / 可回滚

如果业务允许，尽量不要物理删除，而用：

- 软删除；
- 回收站；
- 版本快照；
- 审计日志。

这样即使 Agent 决策错了，也能挽回。

### 第五层：幂等和审计

所有高风险操作都应记录：

- 谁触发；
- 为什么触发；
- 使用了哪个工具；
- 参数是什么；
- 最终执行结果是什么。

### 面试里的收口方式

> 防误删不能寄希望于 prompt 写得够好，而要把系统设计成“默认保守、写操作受控、操作可审计、结果可回滚”。模型只负责建议和编排，真正的高风险执行必须被制度化约束。

## 9、用户提出模糊需求如“按老样子帮我订一下”，Agent 如何处理？

### 参考答案

这种问题本质上是**意图存在，但参数不完整**。Agent 不能自作主张，也不能一味反问，要平衡体验和风险。

我的处理思路一般是：

### 第一步：判断这是低风险还是高风险场景

- 低风险场景：可以基于历史模板先给出候选方案；
- 高风险场景：必须先确认。

比如“订个会议室”和“下采购单”显然风险等级不同。

### 第二步：找“老样子”的来源

可参考的来源通常有：

- 最近一次成功记录；
- 用户的偏好配置；
- 组织默认模板；
- 同类历史操作。

### 第三步：补足缺失参数

如果缺失参数不影响安全，可以自动补；如果影响结果或有风险，要追问。

例如：

> 我找到了你上次的订购记录：仓库 A、供应商 B、数量 200、送达日期本周五。请确认这次是否还是这套配置？

### 第四步：形成“候选 + 确认”而不是直接执行

特别是写操作，最稳妥的是让 Agent 给出一个结构化确认单，而不是直接落单。

### 为什么这样处理

因为“按老样子”其实依赖长期记忆和上下文理解，但真正落地时不能把“相似”当成“正确”。所以我会回答：

> Agent 要做的是把模糊需求转成可执行候选，并在风险点前做确认，而不是把历史经验直接当成确定命令去执行。

### 结合你的业务背景可以加一句

> 在供应链系统里，这种模糊表达特别常见，但业务代价也高，所以我会更强调模板复用 + 参数回显 + 风险前确认。

## 10、如何量化评估一个上线的 Agent 好坏？除了准确率还有哪些指标？

### 参考答案

我不会只看准确率，因为 Agent 不是单轮分类器，而是一个有过程、有工具、有成本的系统。评价我通常分四类。

### 1. 结果指标

- 任务完成率；
- 最终答案正确率；
- 用户接受率；
- 人工接管率；
- 业务目标达成率。

### 2. 过程指标

- 平均工具调用次数；
- 失败重试次数；
- 死循环率；
- Wait 率；
- 平均步骤数；
- 中途退出率。

### 3. 成本指标

- 单任务 token 成本；
- 平均时延 / P95 时延；
- 工具调用成本；
- 检索成本；
- 每成功任务成本。

### 4. 稳定性与安全指标

- 超时率；
- 错误率；
- 高风险操作拦截率；
- 幻觉率；
- 数据越权率；
- 不可恢复失败率。

### 如果再往前一步

我会把 Agent 评估拆成：

- **Offline evaluation**：标准集、回放集、对比集；
- **Online evaluation**：A/B test、人工抽检、用户反馈闭环。

### 我的总结方式

> 评估 Agent 不能只看“答得对不对”，还要看“是不是稳、是不是便宜、是不是安全、是不是完成了业务目标”。真正上线后，我更关注任务完成率、人工接管率、时延、成本和风险指标的组合。

## 11、Kafka 在 AI Agent 系统中可能用于哪些场景？如何保证消息顺序性？

### 参考答案

Kafka 在 Agent 系统里更适合做高吞吐、可回放的事件总线，常见场景包括：

- Agent 执行事件流采集；
- 工具调用日志；
- 观测数据上报；
- 异步任务分发；
- 多 Agent 协作事件流；
- 审计与离线分析。

### 为什么 Kafka 合适

- 吞吐高；
- 可持久化；
- 支持消费组；
- 适合过程日志、事件溯源和离线分析。

### 怎么保证消息顺序性

Kafka 的顺序性本质上是 **分区内有序**，不是全局有序。所以设计时要先确定“你要的顺序范围是什么”。

#### 如果要保证单个 session 内顺序

可以把 `sessionId` 作为 partition key，这样同一个 session 的事件都会落到同一分区，天然保证分区内顺序。

#### 如果要保证单个 task 内顺序

就用 `taskId` 做 key。

### 还要注意什么

1. 一个分区内同一消费组最好单线程消费，才能保证严格顺序；
2. 重试和死信设计不能打乱顺序；
3. 如果某些事件允许并行，那就不要强求全局顺序，不然吞吐会被锁死。

### 我会怎么回答

> Kafka 在 Agent 系统里主要解决高吞吐事件流和异步解耦问题。顺序性要按业务粒度设计，通常不是保证全局顺序，而是保证同一 session、同一 task、同一 agent 维度的局部顺序。

## 12、Elasticsearch 在 RAG 系统中的作用是什么？如何优化检索性能？

### 参考答案

Elasticsearch 在 RAG 里最核心的作用，是做 **关键词检索、倒排索引和混合检索基础设施**。

### 具体作用

1. 做 BM25 关键词召回；
2. 承载 metadata filter，比如租户、知识域、时间范围、标签；
3. 配合向量字段做 hybrid search；
4. 支持高亮、分词、同义词和检索调优。

### 为什么 ES 很重要

因为很多企业知识问题并不是纯语义问题，而是带关键词、术语、版本号、接口名、SKU、单号的，这类问题 ES 很强。

### 如何优化检索性能

#### 1. 索引设计

- 合理拆索引；
- 控制字段数量；
- 分清 text / keyword 字段；
- chunk 大小不要失控。

#### 2. 检索策略优化

- 先 filter 再 recall；
- topK 不要盲目拉太大；
- 热门 query 做缓存；
- 对长 query 先改写。

#### 3. 混合检索优化

- BM25 + vector 结合；
- rerank 时只处理候选集；
- 父子块配合。

#### 4. 集群层优化

- 合理 shard 数；
- 副本数按读写比调；
- 热冷分层；
- 控制 refresh 频率。

### 一句话总结

> 在 RAG 里，ES 的价值不是“只是一个搜索库”，而是企业知识检索的骨架，尤其适合关键词强、过滤条件多、要做混合检索的场景。

## 13、手写算法：计算二叉树最大宽度

### 参考答案

这题常见做法是 BFS，并给每个节点一个“完全二叉树下标”，同层宽度用最后一个位置减第一个位置再加一。

```java
import java.util.LinkedList;
import java.util.Queue;

class TreeNode {
    int val;
    TreeNode left;
    TreeNode right;
    TreeNode(int val) { this.val = val; }
}

class Pair {
    TreeNode node;
    long index;
    Pair(TreeNode node, long index) {
        this.node = node;
        this.index = index;
    }
}

public class Solution {
    public int widthOfBinaryTree(TreeNode root) {
        if (root == null) return 0;
        Queue<Pair> queue = new LinkedList<>();
        queue.offer(new Pair(root, 1L));
        int ans = 0;

        while (!queue.isEmpty()) {
            int size = queue.size();
            long left = queue.peek().index;
            long right = left;

            for (int i = 0; i < size; i++) {
                Pair cur = queue.poll();
                long idx = cur.index;
                right = idx;

                if (cur.node.left != null) {
                    queue.offer(new Pair(cur.node.left, idx * 2));
                }
                if (cur.node.right != null) {
                    queue.offer(new Pair(cur.node.right, idx * 2 + 1));
                }
            }

            ans = Math.max(ans, (int)(right - left + 1));
        }
        return ans;
    }
}
```

### 复杂度

- 时间复杂度：`O(n)`
- 空间复杂度：`O(n)`

### 面试时讲法

> 这题关键不是节点数，而是“位置编号”。只统计每层节点个数会错，因为中间可能有空位，所以要用完全二叉树编号去算宽度。

## 2、从架构角度分析 Tools、Workflow 和 Agent 三者的本质区别

### 参考答案

我会把三者看成三个不同抽象层级：

### 1. Tools：能力原子

Tool 是最小执行单元，本质上是一个**受约束的外部能力接口**。它关注的是：

- 做什么；
- 输入输出是什么；
- 是否有副作用；
- 风险边界是什么。

比如搜索、读文件、执行 shell、发消息、调用业务接口，这些都属于 Tool。

### 2. Workflow：预定义流程

Workflow 是把多个步骤按确定性逻辑串起来，本质上是**流程编排**。它更强调：

- 固定顺序；
- 明确分支；
- 状态可控；
- 可预测性强。

比如“先检索、再总结、再审批、最后通知”就是 Workflow。

### 3. Agent：带目标驱动的动态决策体

Agent 在 Tools 和 Workflow 之上，多了“根据上下文做动态决策”的能力。它不只是执行固定流程，而是会决定：

- 下一步要不要用工具；
- 用哪个工具；
- 是否需要问用户；
- 是否更新计划；
- 是否该停止。

### 本质区别怎么一句话说清楚

- **Tool**：能做什么；
- **Workflow**：按什么流程做；
- **Agent**：为什么这么做、下一步该怎么做。

### 结合 `mooc-manus-java` 举例

- 文件、Shell、浏览器、搜索等是 Tool；
- `SpringAIPlannerReActFlow` 是一种带状态流转的流程编排；
- Planner + ReAct 组合在目标驱动下动态决定步骤更新和工具调用，这才接近 Agent。

### 我认同的架构观点

> 很多系统其实并不需要一上来就做 Agent。能用 Workflow 解决的，不一定非要 Agent；能用 Tool 直接做的，也没必要引入复杂决策层。成熟架构应该是 Tool、Workflow、Agent 三层并存，而不是只有 Agent。

## 3、Multi-Agent 系统如何设计三层架构？Agent 之间如何通信？

### 参考答案

如果做 Multi-Agent，我比较认同三层结构：

### 第一层：能力层（Capability Layer）

这一层沉淀基础能力：

- Tool；
- Memory；
- RAG；
- 安全控制；
- 任务队列；
- 观测系统。

所有 Agent 都共享这层基础设施。

### 第二层：协作层（Coordination Layer）

这一层负责 Agent 之间的协作关系，例如：

- 主 Agent / 子 Agent 调度；
- 任务拆分；
- 路由分发；
- 状态同步；
- 冲突解决；
- 结果聚合。

### 第三层：角色层（Role Layer）

这一层是具体 Agent 角色，比如：

- Planner Agent；
- Executor Agent；
- Retriever Agent；
- Reviewer Agent；
- Domain Agent。

### Agent 之间如何通信

我觉得有三种主流方式：

#### 1. 共享状态通信

通过共享 memory、共享任务上下文、共享状态仓库来通信。

优点是简单；缺点是耦合高。

#### 2. 消息通信

通过事件总线、MQ、消息 envelope 来传递任务和结果。这个更适合分布式和异步场景。

#### 3. 协议通信

通过 A2A、MCP、HTTP API 等标准协议，让一个 Agent 以“调用另一个 Agent 的服务”的方式协作。

### 我更偏向哪种

如果是单机或单服务内协作，可以共享状态 + 内部调用；如果是跨服务、跨团队、跨运行时协作，我会更偏向协议化和消息化通信。

### 一句话总结

> Multi-Agent 的难点不在于“多几个模型实例”，而在于角色划分、协作协议、状态一致性和故障隔离。

## 4、当前阻碍 Agent 大规模落地的最大挑战是什么？如何解决可控性和能力的平衡问题？

### 参考答案

我觉得当前最大挑战不是单点能力，而是 **可控性、稳定性和成本三者之间的平衡还不够好**。

具体看有几个难点：

### 1. 可控性不足

Agent 天生有开放式决策空间，很难像传统程序那样完全预测。

### 2. 稳定性不够

同样的问题，模型输出可能波动；工具链一长，失败点也更多。

### 3. 成本高

长上下文、多轮推理、RAG、工具调用都会放大 token 和时延成本。

### 4. 安全边界难定义

特别是一旦 Agent 开始接业务写操作，权限和风控问题会迅速放大。

### 怎么平衡可控性和能力

我的观点是：**不要把可控性和能力看成非此即彼，而是要通过分层架构让高能力只在必要场景下释放。**

我会这样做：

#### 第一，明确层级

- Tool 层强约束；
- Workflow 层半开放；
- Agent 层只处理真正需要动态决策的部分。

#### 第二，状态和事件外显

不要让 Agent 黑盒运行，要能看到它当前 plan、step、tool call 和终止原因。

#### 第三，高风险操作加制度化约束

如审批、白名单、dry-run、回滚。

#### 第四，按任务分级

- 简单任务：规则 / workflow；
- 中等任务：单 Agent；
- 高复杂任务：多 Agent / ToT / 更强策略。

### 结论表达

> 现在阻碍 Agent 大规模落地的，不是“模型不够聪明”这么简单，而是工程系统还没有把能力、安全、稳定性和成本协同到一个平衡点上。真正的解法不是一味增强模型，而是架构分层、能力分级和风险收口。

## 5、MCP、A2A、Skills 和 Function Call 分别是什么？它们之间有什么区别？

### 参考答案

这几个概念很容易混，但它们所在层次不一样。

### 1. Function Call

Function Call 是模型调用外部能力的一种接口机制。模型输出结构化参数，系统再去执行对应函数。

它更像是：**模型和工具之间的调用协议。**

### 2. Skills

Skills 可以理解为更高层的“封装能力块”，它可能由多个工具、prompt、规则甚至子流程组成，对外暴露一个更稳定的能力单元。

它更像是：**可复用的业务能力组件。**

### 3. MCP

MCP 是 Model Context Protocol，本质上是在模型 / Agent 与外部工具、资源、上下文之间建立一个更标准化的协议层。

它关注的是：

- 如何发现能力；
- 如何描述能力；
- 如何按统一协议调用。

在 `mooc-manus-java` 里也能看到 `McpPort`、`McpToolCallbackProvider` 这类设计，说明项目已经考虑把外部工具能力标准化接入。

### 4. A2A

A2A 是 Agent-to-Agent，强调的是 Agent 之间按某种协议互相协作。

它不是工具调用，而是：

- 一个 Agent 把另一个 Agent 当作协作对象；
- 传任务、收结果；
- 可能带角色能力声明。

`mooc-manus-java` 里有 `A2APort` 和相关 provider，这就很适合拿来说明它和普通 Function Call 的区别。

### 这四者的区别怎么概括

- **Function Call**：模型调用工具的机制；
- **Skill**：对工具和流程的高层封装；
- **MCP**：模型 / Agent 与外部能力的标准化协议；
- **A2A**：Agent 与 Agent 之间的协作协议。

### 面试里一句话说法

> Function Call 解决“怎么调一个函数”，Skill 解决“怎么复用一段能力”，MCP 解决“怎么标准化接外部能力”，A2A 解决“多个 Agent 之间怎么协作”。

## 6、Agent 的上下文工程有哪些主流技术？如何设计有效的上下文管理策略？

### 参考答案

上下文工程是我觉得 Agent 落地里最关键的部分之一，因为很多问题最后不是模型能力不够，而是上下文喂得不对。

主流技术我会分成几类：

### 1. Prompt 分层

把 system、developer、task instruction、tool schema、memory、retrieval context 分层，不要全揉在一块。

### 2. Recent Window

保留最近几轮原始消息，维持当前对话连续性。

### 3. Dynamic Summary

把老上下文压缩成 summary，减少 token 压力。

### 4. Retrieval Augmentation

按需检索知识，而不是把所有知识塞进 prompt。

### 5. Structured State

把关键事实、目标、约束、当前阶段抽成结构化状态，而不是依赖模型自己从文本中恢复。

### 6. Tool / Role Context Isolation

不同角色、不同工具上下文隔离，比如 planner 和 executor 分开记忆，避免污染。

`mooc-manus-java` 的 `AgentMemoryStore` 本身就是按 `sessionId + agentName` 隔离的，这个点很值得讲。

### 我会怎么设计上下文管理策略

#### 第一，确定上下文组成

一般包括：

- 系统规则；
- 当前用户输入；
- 最近窗口；
- 关键摘要；
- 当前计划与状态；
- 检索上下文；
- 工具定义。

#### 第二，按角色和任务裁剪

planner 需要全局目标，executor 更需要当前步骤和相关 observation，不同 Agent 上下文不应一样。

#### 第三，控制预算

每部分都要有 token budget，尤其是 retrieval 和 memory，不能无限长。

#### 第四，做回放和评估

上下文工程不是拍脑袋，要通过 trace 回放看：

- 哪部分真正有用；
- 哪部分污染模型；
- 哪部分最耗 token。

### 我认可的总结

> 上下文工程的核心不是“塞更多”，而是“只在当前时刻塞最相关、最可靠、最可控的信息”。

## 13、手写算法：LeetCode 22，生成括号组合

### 参考答案

这题经典做法是回溯。核心约束是：

- 左括号数量不能超过 `n`；
- 右括号数量不能超过左括号数量。

```java
import java.util.ArrayList;
import java.util.List;

public class Solution {
    public List<String> generateParenthesis(int n) {
        List<String> ans = new ArrayList<>();
        backtrack(ans, new StringBuilder(), 0, 0, n);
        return ans;
    }

    private void backtrack(List<String> ans, StringBuilder path, int left, int right, int n) {
        if (path.length() == 2 * n) {
            ans.add(path.toString());
            return;
        }

        if (left < n) {
            path.append('(');
            backtrack(ans, path, left + 1, right, n);
            path.deleteCharAt(path.length() - 1);
        }

        if (right < left) {
            path.append(')');
            backtrack(ans, path, left, right + 1, n);
            path.deleteCharAt(path.length() - 1);
        }
    }
}
```

### 面试讲解

- `left` 表示已放的左括号数；
- `right` 表示已放的右括号数；
- 只要 `left < n`，就还能放左括号；
- 只要 `right < left`，就还能放右括号。

### 复杂度

结果数量是卡特兰数级别，时间复杂度通常写成与结果数量相关，近似 `O(Cn)`。
